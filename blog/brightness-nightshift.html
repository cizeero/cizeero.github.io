Brightness to Night Mode

I run Sway on Fedora. I already had a nightshift script that shifts my display into warm amber tones using wl-gammarelay-rs; low mode at 4000K, high mode at 2500K where everything becomes near-monochrome amber, I used this almost every night. But I kept noticing the friction of manually running nightshift low when I have already hit 0% brightness. Two separate controls for what is really one continuous action, less light. So I made it into one. When brightness hits 0% and you press the brightness-down key again, it enters nightshift low and then nightshift high.

The implementation is a 80-line bash script that sits between the brightness keys and brightnessctl. It checks the current color temperature via DBus to know which mode it's in, and decides what to do next. The sway config override is 2 lines. I never read the script, it purely written via claude code, first time it bugged, the next time it worked and I haven't touched it since. The entire display pipeline is yours, do whatever you want with it. wl-gammarelay-rs exposes gamma, temperature, and brightness as DBus properties. brightnessctl talks to the kernel backlight interface. Sway lets you rebind any key to any commands, the actual interfaces designed to be composed. Wire them together and you get something for yourself, on a personal level. A couple more effort and I can have schedule working in background, with just one more claude code command, but my non existent schedule prompts to a non existent nightshift cycle.

Its a real argument for Linux, because that it treats the computer as a thing you can completely modify. All the layers are accessible. The cost is that you receive an unpolished experience but that's the payoff because you polish it yourself
